1/13 Introdução

Instruções

Cada passo no tutorial inicia com um texto. No final de cada texto há um botão que inicia a execução do comando git.

Após clicar no botão 4 coisas vão acontecer, na seguinte sequência:

  • No console (no canto inferior esquerdo): O comando será adicionado e executado.
  • À direita: Uma animação mostrará o que o comando faz.
  • Quando a animação termina, o console mostra a saída do comando.
  • À direita do botão aparecerá um link para ir para a próxima etapa.

A definição oficial de git é "um sistema de controle de versão distribuído". Mas o que isto significa? Vamos usar um exemplo para explicar isso.

Na tabela abaixo temos algumas mudanças que fizemos em um arquivo:

Tempo Status
07 de julho, 10:00 Arquivo tasks.txt criado.
07 de julho, 00:05 Adicionar paragrafo para o arquivo tasks.txt.
07 de julho, 10:10 Formatar o conteúdo de tasks.txt.
... ...

Podemos supor que cada uma das linhas na tabela é uma versão do arquivo. Então o git é um sistema que nos ajuda a rastrear a versão ou alterações de arquivos ao longo do tempo (ele também é distribuído, mas vamos deixar isso para outro tutorial).

Conceitos básicos do Git

Para inicializar o git em uma pasta, usaremos o comando git init. Depois de executar este comando, o git vai começar a monitorar mudanças dentro desta pasta.

Internamente o git têm três estados (também chamados de áreas) onde as mudanças podem ser feitas: diretório de trabalho, área de testes e repositório local.

Clique no botão abaixo para iniciar a animação.

git init Próximo

2/13 Primeiras Mudanças

A primeira coisa que precisamos para ver o git em ação, é um arquivo. Podemos criar um novo arquivo com o comando touch e passando um nome de arquivo como argumento (este é não é um comando git e sim UNIX).

Este comando irá criar o arquivo tasks.txt. Neste arquivo, vamos manter uma lista de tarefas que precisamos fazer.

O diretório de trabalho rastreia todas as mudanças que fazemos dentro da pasta. Então o git irá mostrar esse novo arquivo dentro desta área.

touch tasks.txt Próximo

3/13 Adicionando alterações para a área de testes

Um commit é uma mudança ou um grupo de mudanças, nós envolvemos em conjunto e dizemos para o git salvar. Ele é criado em 2 etapas: a primeira é a adição de todas as mudanças que fizermos para a área de testes.

Para adicionar as mudanças para a área de teste usamos o comando git add. O próximo comando move as mudanças do arquivo tasks.txt do diretório de trabalho para a área de teste.

git add tasks.txt Próximo

4/13 Criando nosso primeiro commit

Anteriormente, vimos a primeira das 2 etapas para criar um commit.

O segundo passo é dizer explicitamente para o git envolver todas as alterações atuais na área de teste em um commit. Nós podemos fazer isso com o comando git commit.

Damos cada commit uma mensagem para identificar facilmente o seu conteúdo mais tarde. Para passar uma mensagem para o git commit usamos a opção -m.

O próximo comando irá criar o nosso primeiro commit com a mensagem Add file tasks.txt .

git commit -m "Add file tasks.txt" Próximo

5/13 Modificando um arquivo

O comando echo misturado com >> adiciona linhas ao final de um arquivo.

A primeira tarefa que nós iremos fazer é Comprar Leite. Então vamos adicioná-lo ao nosso arquivo tasks.txt com o comando echo.

Veja como o git nos diz que o arquivo foi alterado, adicionando-o para o diretório de trabalho.

echo "Buy Milk" >> tasks.txt Próximo

6/13 Git Status

Outro comando útil é o git status. Este comando nos mostra as mudanças que temos dentro do diretório de trabalho e a área de teste.

Olhe para a saída do comando no console (canto inferior esquerdo). O git status vai nos dizer que as mudanças não foram organizadas (isso significa que as mudanças estão dentro do diretório de trabalho).

Todos os arquivos com alterações dentro do diretório de trabalho serão mostradas em vermelho. E o arquivo tasks.txt tem um status modificado porque nós adicionamos uma linha para ele na última etapa.

git status Próximo

7/13 Detalhando Mudanças

O comando git status apenas nos mostra os nomes dos arquivos com alterações. Se quisermos saber o que mudou dentro de cada arquivo do diretório de trabalho, podemos usar o comando git diff.

Na saída do git diff, as linhas que adicionamos ao arquivo são mostrados em verde, e os que foram removidos são mostrados em vermelho.

git diff Próximo

8/13 Add novamente

Nós queremos criar um outro commit com as alterações que fizemos em tasks.txt. Nós precisamos adicionar essas mudanças para a área de testes novamente com o comando git add.

git add tasks.txt Próximo

9/13 Git status 2

Podemos usar o comando git status novamente para ver a mensagem que o git nos mostra quando fazemos as mudanças. O arquivo tasks.txt ainda aparece com um status modificado, mas a cor mudou para verde, o que significa que as alterações estão dentro da área de testes.

git status Próximo

10/13 Detalhando Mudanças 2

Na etapa 7 foi utilizado o comando git diff para ver a lista detalhada das mudanças que tivemos dentro do diretório de trabalho.

Este comando também pode mostrar as mudanças dentro da área de testes, se usarmos a opção --cached nele.

A saída tem o mesmo código de cores: as linhas que adicionamos são mostrados em verde, e os que nós removemos são mostrados em vermelho.

git diff --cached Próximo

11/13 Nosso Segundo Commit

Queremos que as mudanças na área de teste sejam commitadas. Usaremos o comando git commit novamente, e passaremos uma mensagem descritiva para este commit.

git commit -m "Adicionar primeira tarefa" Próximo

12/13 Ver os Nossos Commits

O comando que completa este tutorial é git log. Este comando nos mostra os commits que fizemos.

O git log tem várias opções para mostrar na saída com diferentes níveis de detalhe. Usamos a opção --oneline para ver a saída em um formato curto.

git log --oneline Próximo

13/13 Encerramento

Este tutorial tem todos os comandos básicos que você precisa para começar a usar git em seus projetos pessoais.
